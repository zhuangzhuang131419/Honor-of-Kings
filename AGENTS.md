# Honor of Kings 对局记录与分析项目指南

## 项目目标

本项目用于记录朋友之间的《王者荣耀》对局，并基于对局数据做胜率、表现、PK 和排行榜分析。第一版目标是做出可部署到 GitHub Pages 的静态前端 MVP，让管理员可以维护 CSV 数据，普通访问者可以查看分析页面。

## 技术栈与部署

- 前端使用 React + Vite + TypeScript。
- 应用以静态站点形式部署到 GitHub Pages。
- 第一版不创建后端、不引入数据库、不实现真实鉴权。
- 所有正式公开数据来自仓库内的 `public/data/matches.csv`。
- 管理员通过网页导入 CSV、手动录入草稿或合并导出 CSV，再提交到仓库更新公开数据。

## 数据方案

`public/data/matches.csv` 是第一版唯一正式数据源。CSV 使用“一名玩家一行”的结构，一局完整对局应包含双方 10 名玩家，也就是 10 行记录。

建议字段如下：

- 对局字段：`matchId`, `playedAt`, `mode`, `durationSeconds`, `blueKills`, `redKills`, `winningSide`
- 玩家字段：`side`, `summoner`, `hero`, `position`, `rating`, `kills`, `deaths`, `assists`
- 表现字段：`damageDealt`, `damageTaken`, `gold`, `teamfightRate`
- 扩展字段：`isMvp`, `medal`, `item1`, `item2`, `item3`, `item4`, `item5`, `item6`, `notes`

字段约定：

- `side` 使用 `blue` 或 `red`。
- `winningSide` 使用 `blue` 或 `red`。
- `playedAt` 使用可排序日期时间格式，例如 `2026-06-09 22:31` 或 ISO 时间。
- KDA 拆为 `kills`, `deaths`, `assists` 三个数字字段。
- 百分比字段如 `teamfightRate` 建议存为 `0-100` 的数字。

## 核心功能

### 对局录入

- 支持手动录入双方 10 人数据。
- 支持导入外部 CSV，展示记录数、对局数和结构校验提示。
- 每名玩家需要记录召唤师、英雄、位置、评分、KDA、输出、承伤、经济、参团率、阵营和胜负。
- 录入数据先进入可编辑草稿，确认后可导出为与 `matches.csv` 字段一致的 CSV。

### 截图识别暂缓

- 第一版先放弃网页端 OCR 录入，统一以 CSV 作为导入、校验、编辑和发布格式。
- 外部截图可以先通过人工或其它离线方式整理成符合字段约定的 CSV，再导入页面校验。
- 后续如果恢复 OCR，只能生成可编辑草稿，不得绕过人工确认直接写入正式数据。
- 一局完整截图通常由两张图拼成：总览/KDA 图和数据面板图；这一信息仅作为未来增强参考。

### 分析页

- 提供筛选条件：召唤师、英雄、队友、位置、模式、胜负、日期范围。
- 筛选条件应联动：选择召唤师后，英雄、队友、位置等候选项应根据当前数据范围收窄；切换其它条件也应同步影响可选项。
- 队友筛选支持多选，多选时表示目标召唤师同阵营同时包含这些队友。
- 顶部提供时间选择，支持日、周、月维度。
- 中间展示胜率柱状图，横坐标为时间，纵坐标为胜率。
- 下方展示分析表格，包含场次、胜率、平均评分、KDA、输出、承伤、经济、MVP 次数等指标。
- 表格支持下钻到单局详情，展示双方 10 人的核心数据。

### PK 页

- 支持选择两个召唤师进行对比。
- 每个召唤师可附加英雄、队友、位置、日期范围等筛选条件。
- 对比指标包括胜率、场次、平均评分、KDA、输出、承伤、经济和 MVP 次数。
- 两侧筛选条件应相互独立，避免互相污染统计结果。

### 排行榜

- 支持日、周、月维度。
- 按召唤师展示胜率排行。
- 展示字段包括排名、召唤师、场次、胜率、平均评分、KDA、MVP 次数。
- 应提供最低场次门槛，避免少量对局造成榜单失真。

## 统计口径

- “召唤师”按游戏昵称统计。
- 胜率按该召唤师所在阵营是否等于 `winningSide` 计算。
- 队友筛选表示同一局、同一阵营中包含指定召唤师。
- 英雄筛选只作用于目标召唤师自己的英雄。
- 位置筛选只作用于目标召唤师自己的位置。
- 日、周、月聚合基于 `playedAt`。
- KDA 展示可使用 `(kills + assists) / max(1, deaths)`。

## UI 与交互风格

- 整体采用暗色电竞风格和玻璃拟态质感。
- 使用深蓝背景，蓝色代表蓝方，红色代表红方。
- 图表、关键指标和导航可以使用轻微发光效果，但要保证可读性。
- 页面应是工具型应用，不做营销式首页。
- 首屏应直接进入数据分析或主要工作台。
- 布局保持紧凑、清晰，适合反复筛选、比较和查看表格。
- 卡片圆角控制在 8px 左右，避免过度圆润。
- 桌面端优先展示完整表格和图表；移动端需要保证筛选器、图表和表格不重叠。

## 测试要求

实现功能时至少覆盖以下验证场景：

- CSV 解析：空文件、缺字段、数字格式、日期格式、10 人完整对局、缺少部分玩家的对局。
- 统计口径：单召唤师胜率、英雄筛选、队友筛选、位置筛选、日/周/月聚合。
- PK 页面：双方筛选条件互不影响，对比指标计算正确。
- 排行榜：最低场次门槛生效，胜率和排序正确。
- 录入流程：必填字段校验、KDA 数值校验、胜负阵营校验。
- CSV 导入流程：文件上传、结构校验、导出导入 CSV、合并导出后重新导入一致。
- UI 响应式：桌面和移动端无文字溢出、重叠或图表不可读。
- GitHub Pages：静态构建后能正确读取 `public/data/matches.csv`。

## 实现边界

- 不要在第一版中加入后端服务、数据库或真实登录鉴权。
- 第一版不要实现网页端 OCR 录入；如后续恢复，也不要让 OCR 结果绕过人工确认直接成为正式数据。
- 不要把正式数据写入浏览器本地存储作为唯一来源。
- 不要创建与 CSV 字段不兼容的导出格式。
- 如需新增字段，应保持向后兼容，并更新 CSV 示例、解析逻辑和测试。
